## §1 ABSOLUTE PROHIBITIONS (ZERO TOLERANCE)

These are not guidelines. These are laws. Violation of ANY of these results in immediate rejection.

### §1.1 MINIMIZE `#[allow(...)]` IN HAND-WRITTEN CODE

```rust
// FORBIDDEN without justification
#[allow(clippy::unwrap_used)]
#[allow(dead_code)]
#[allow(unused_variables)]
#![allow(clippy::pedantic)]
```

If clippy or rustc emits a warning in **hand-written source**, **prefer fixing the underlying issue**.

**Genuine false positives:** If a lint is a documented false positive, you may use `#[allow]` with an inline comment explaining why:

```rust
// Clippy false positive: https://github.com/rust-lang/rust-clippy/issues/XXXX
#[allow(clippy::some_lint)]
fn example() { ... }
```

**"But this is autogenerated code"** — Generated code is not an engineering surface. Isolate it and suppress lints at the boundary.

**"But pedantic is too strict"** — It is; `clippy::pedantic` includes lints that are "rather strict or have occasional false positives". These are warnings, not errors.

#### Allowed carve-out: generated code boundary modules ONLY

`#[allow]` / `#![allow]` is permitted only on a tiny, dedicated module that contains **only generated code** (e.g., `include!(concat!(env!("OUT_DIR"), "..."))` or `tonic::include_proto!`) and **no hand-written items**.

```rust
#[allow(clippy::all, clippy::pedantic, clippy::nursery)]
pub mod pb {
    include!(concat!(env!("OUT_DIR"), "/pb.rs"));
}
```

If you use protobuf generators, these APIs exist for attribute injection into generated types:

- `prost_build::Config::type_attribute` (docs.rs: `https://docs.rs/prost-build/latest/prost_build/struct.Config.html`)
- `tonic_prost_build::Config::type_attribute` (docs.rs: `https://docs.rs/tonic-prost-build/latest/tonic_prost_build/struct.Config.html`)

### §1.2 NO Borrow-Checker Appeasement

These patterns indicate you don't understand ownership. They are banned:

| Banned Pattern | Why It's Banned |
|----------------|-----------------|
| `.clone()` to satisfy borrow checker | You're hiding a design flaw |
| `Rc<RefCell<T>>` across module boundaries | Runtime panics, spaghetti ownership |
| `Arc<Mutex<T>>` for non-concurrent data | Lying about your concurrency model |
| Lifetime annotations on every struct | Over-borrowed design, rethink data flow |
| `'static` bounds to avoid thinking | Prevents borrowing, forces allocation |
| `Box<dyn Trait>` without justification | Prefer generics; use `dyn` for heterogeneous collections or plugin systems |

**If borrowing is hard, your data model is wrong. Redesign it.**

### §1.3 NO Silent Failures

| Banned | Why |
|--------|-----|
| `.unwrap()` (non-test code) | Crashes caller's program on None/Err |
| `.expect("msg")` (non-test code) | Same, with slightly better panic message |
| `let _ = fallible_operation();` | Discarding errors silently |
| `if let Ok(x) = result { ... }` without else | Ignoring the error case |
| `match result { Ok(x) => x, Err(_) => return }` | Throwing away error information |

Tests are different: panics are a valid failure signal.

If the repository has `clippy.toml` / `.clippy.toml`, you may allow unwrap/expect in tests via Clippy config:

```toml
# clippy.toml
allow-unwrap-in-tests = true
allow-expect-in-tests = true
```

Source: Clippy lint configuration (`https://doc.rust-lang.org/clippy/lint_configuration.html`).

**Every Result must be:**

- Propagated with `?` (and the error type must be meaningful)
- Handled with explicit match arms that DO something with the Err
- Converted to Option only if absence is semantically equivalent to failure

## §3 TYPE-DRIVEN DESIGN (MANDATORY)

### §3.1 Parse, Don't Validate

Raw primitives (`String`, `i32`, `bool`, `Vec<u8>`) are for I/O boundaries ONLY.

```rust
// ❌ BAD: Primitives flowing through the system
fn process_user(name: String, age: i32, email: String) -> Result<(), Error>

// ✅ GOOD: Parsed newtypes with validated invariants
fn process_user(user: ValidatedUser) -> Result<(), ProcessError>

struct ValidatedUser {
    name: NonEmptyString,    // Guarantees non-empty
    age: Age,                // Guarantees 0-150 range
    email: EmailAddress,     // Guarantees valid format
}
```

**Every newtype must have:**

- A fallible constructor (`new`, `try_from`, `parse`) that validates
- No way to construct an invalid instance
- `#[derive(Debug, Clone, PartialEq, Eq)]` at minimum

### §3.2 Make Invalid States Unrepresentable

```rust
// ❌ BAD: Invalid states are representable
struct Connection {
    socket: Option<TcpStream>,
    is_connected: bool,        // Can be true when socket is None!
}

// ✅ GOOD: State encoded in type system
enum Connection {
    Disconnected,
    Connected { socket: TcpStream },
}
```

### §3.3 Typestate for Protocols

```rust
// ❌ BAD: Runtime state checking
impl Builder {
    fn build(&self) -> Result<Thing, Error> {
        if self.required_field.is_none() {
            return Err(Error::MissingField);  // Runtime error!
        }
        // ...
    }
}

// ✅ GOOD: Compile-time state enforcement
impl Builder<NoName> {
    fn name(self, n: String) -> Builder<HasName> { ... }
}
impl Builder<HasName> {
    fn build(self) -> Thing { ... }  // Can only be called after name()!
}
```

## §4 ERROR HANDLING (EXPLICIT AND MEANINGFUL)

### §4.1 Error Type Requirements

Every error type must:

1. Be an enum with specific variants (not `struct Error(String)`)
2. Implement `std::error::Error`
3. Have meaningful `Display` messages that help the USER (not the developer)
4. Include context about what operation failed and why

```rust
// ❌ BAD
#[derive(Debug)]
struct Error(String);

// ❌ ALSO BAD
enum Error {
    Io(std::io::Error),
    Parse(ParseError),
    Other(String),  // Lazy catch-all!
}

// ✅ GOOD
#[derive(Debug, thiserror::Error)]
enum ConfigError {
    #[error("failed to read config file '{path}': {source}")]
    ReadFailed {
        path: PathBuf,
        #[source]
        source: std::io::Error,
    },
    #[error("invalid config at line {line}: {message}")]
    ParseFailed {
        line: usize,
        message: String,
    },
    #[error("missing required field '{field}' in section '{section}'")]
    MissingField {
        section: String,
        field: String,
    },
}
```

### §4.2 Error Propagation

- `?` is for propagation, not for ignoring
- Every `?` site should make sense in the error chain
- If error types don't align, use `.map_err()` to add context, not to erase it

```rust
// ❌ BAD: Erasing error context
let data = std::fs::read(&path).map_err(|_| Error::IoError)?;

// ✅ GOOD: Preserving and adding context
let data = std::fs::read(&path)
    .map_err(|e| ConfigError::ReadFailed { path: path.clone(), source: e })?;
```

## §5 UNSAFE (LAST RESORT, MAXIMUM ISOLATION)

### §5.1 When Unsafe is Permitted

- FFI calls to C libraries (unavoidable)
- Performance-critical code WITH benchmark evidence
- Implementing low-level abstractions (allocators, atomics)

### §5.2 Requirements for Every Unsafe Block

```rust
// SAFETY: [exact invariant being upheld]
// PROOF: [why this invariant holds at this call site]
// TESTED: [how this is tested - miri, fuzzing, etc.]
unsafe {
    // Minimal code here — one operation if possible
}
```

### §5.3 Unsafe Must Be Isolated

```rust
// ❌ BAD: Unsafe exposed in public API
pub unsafe fn dangerous_operation(ptr: *mut u8) { ... }

// ✅ GOOD: Safe wrapper with precondition validation
pub fn safe_operation(data: &mut [u8]) -> Result<(), Error> {
    // Validate preconditions here
    // SAFETY: preconditions validated above
    unsafe { dangerous_inner(data.as_mut_ptr()) }
}
```

## §7 DOMAIN-SPECIFIC ADDENDA

### §7.A Systems / CLI / Server (Default)

- Async: `tokio` for I/O, `rayon` for CPU parallelism
- Error: `thiserror` for libraries, `anyhow` for binaries
- Serialization: `serde` with explicit derives, no `#[serde(default)]` without reason
- CLI: `clap` with derive macros
- Logging: `tracing` with structured fields

### §7.B Embedded / Real-Time / no_std

- Core: `#![no_std]`, `#![no_main]`, `panic = "abort"`
- Memory: NO heap allocation in hot paths. Use `heapless`, `static_cell`, stack allocation.
- Async: `embassy` with static executor
- Real-time: WCET analysis for critical sections. Priority ceiling for shared resources.
- Hardware: `svd2rust` PACs or HALs only. No raw pointer arithmetic without PAC.

### §7.C WebAssembly / Frontend

- Framework: `leptos` or `dioxus` (signals, fine-grained reactivity)
- Interop: `wasm-bindgen` with minimal JS surface
- Size: `opt-level = "z"`, `lto = true`, `codegen-units = 1`
- No `panic!` in WASM — use Result everywhere

### §7.D FFI / Interop

- All shared types: `#[repr(C)]`
- All pointers: validate null, validate alignment, validate length
- All calls: wrap in safe Rust API that enforces preconditions
- Consider `cxx` for C++, `pyo3` for Python
